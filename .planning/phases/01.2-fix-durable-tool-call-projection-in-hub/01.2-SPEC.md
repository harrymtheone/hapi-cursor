# Phase 01.2: Fix durable tool call projection in Hub — Specification

**Created:** 2026-05-25
**Ambiguity score:** 0.11 (gate: ≤ 0.20)
**Requirements:** 6 locked

## Goal

Web tool activity cards show the correct tool name, input, status, and result after refresh, SSE reconnect, or pagination — even when the loaded message window contains only a `tool-call-result` (or legacy `tool-result`) and no matching `tool-call` start — by sourcing identity from a Hub-maintained canonical tool call projection keyed by `callId`, not from in-window start/result pairing alone.

## Background

**Current Web behavior:** `reduceTimeline` in `web/src/chat/reducerTimeline.ts` builds `ToolCallBlock`s by pairing `tool-call` starts with `tool-result` / `tool-call-result` completions inside the loaded message window. When a result arrives without a prior start in that window, `ensureToolBlock` seeds the card with `name: permissionEntry?.toolName ?? 'Tool'` and `input: permissionEntry?.input ?? null` (`reducerTimeline.ts` ~315–326). Permission metadata from `agentState` only helps when Hub still has matching permission entries; it does not carry full tool identity for completed historical tools.

**Current CLI wire behavior:** `cli/src/agent/messageConverter.ts` emits `tool-call` with `name`, `callId`, `input`, and `status`, but `tool-call-result` with only `callId`, `output`, and `is_error`. `web/src/chat/normalizeAgent.ts` maps `tool-call-result` to a single-message `tool-result` content block with no embedded name/input.

**Trigger:** Live bug — Tool activity shows generic `Tool` labels after pagination, page refresh, SSE reconnect, message-window trimming, or event conversion that drops the start side while retaining the result side.

**Gap:** Hub persists raw/standardized agent messages today but does not maintain a merged, session-scoped tool call projection. Web cannot reconstruct complete tool cards from result-only visibility.

## Requirements

1. **Hub canonical projection by callId**: Hub maintains one merged tool call record per session per `callId`.
   - Current: Messages are stored individually; no durable merged tool call state exists in Hub.
   - Target: On ingest of `tool-call` and/or `tool-call-result` agent payloads, Hub upserts a projection row/map entry keyed by `(sessionId, callId)` that merges start and result fields over time.
   - Acceptance: After Hub processes a `tool-call` then `tool-call-result` for the same `callId`, a single projection exists with merged fields; a Hub unit/integration test asserts upsert and merge semantics.

2. **Projection field completeness**: Completed projections carry everything Web tool cards need.
   - Current: Result-only wire events lack `name` and `input`; Web falls back to `'Tool'`.
   - Target: For each `callId`, projection includes at minimum: `callId`, tool `name`, `input`, `status` (`pending` | `in_progress` | `completed` | `failed`), `result` or `error` payload, `startedAt` / `createdAt`, and `completedAt` when known; permission-related metadata is included when Hub already has it for that `callId`.
   - Acceptance: Given stored `tool-call` + `tool-call-result` messages for `callId`, projection JSON/schema validation passes and includes all listed fields with non-empty `name` for a known tool (e.g. `Bash`, `Read`).

3. **Raw message preservation**: Existing message persistence and delivery paths remain intact.
   - Current: Hub stores and serves standardized agent messages as today; Web normalizes them into timeline blocks.
   - Target: Projection is additive — raw/standardized messages continue to be stored and emitted unchanged; projection updates do not replace or delete message history.
   - Acceptance: Existing `hub/src/sync/sessionModel.test.ts` tool-call message tests still pass without modification to expected message storage behavior; new projection logic is covered by additional tests only.

4. **Web result-only rendering**: Web renders correct tool cards when the visible window has only the result-side message.
   - Current: `reduceTimeline` creates placeholder cards with `name: 'Tool'` when no in-window start exists.
   - Target: When building timeline blocks, Web resolves `callId` against Hub-provided projection (via message enrichment, parallel fetch, or SSE patch — implementation in discuss-phase) and seeds `ensureToolBlock` with projection `name`/`input`/`status` instead of the `'Tool'` placeholder.
   - Acceptance: A Web test feeds a message list containing only a normalized `tool-result` for `callId` X plus projection data for X; rendered `ToolCallBlock` shows the real tool name and input, not `'Tool'`.

5. **Lifecycle durability**: Projection-backed rendering survives refresh, SSE reconnect, and pagination.
   - Current: After reload/reconnect/pagination, result-only windows reproduce generic `Tool` cards.
   - Target: Projection is durable in Hub store (survives Hub restart) and available when Web reloads a session page, reconnects SSE, or loads an older page of messages that omits the original `tool-call` start.
   - Acceptance: Test or scripted scenario: (a) ingest start+result, (b) simulate Web load of result-only window with projection fetch — card shows correct name; (c) after repository reload / session refresh path, same assertion holds.

6. **No regression on happy path and grouping**: Normal start+result and grouped Tool activity unchanged.
   - Current: In-window `tool-call` + `tool-result` pairing and `web/src/chat/toolGroups.ts` grouping work for live sessions.
   - Target: When both start and result are present in the window, rendered cards match pre-fix behavior (names, inputs, results, timestamps, Task/Agent sidechain children, grouped tool activity summaries).
   - Acceptance: Existing `web/src/chat/reducerTimeline.test.ts`, `web/src/chat/toolGroups.test.ts`, and `web/src/components/ToolCard/*.test.tsx` suites pass without intentional snapshot/behavior changes except tests explicitly updated for projection wiring.

## Boundaries

**In scope:**
- Hub-side merge/upsert of tool call state keyed by `callId` per session
- Durable persistence of projection in Hub store (SQLite), following existing session-scoped durability patterns
- Exposing projection to Web on session message load and/or live updates (exact transport decided in discuss-phase)
- Web timeline/reducer changes to consume projection for result-only and partial windows
- Shared protocol types/schemas for projection payloads in `@hapi/protocol` if needed
- Regression tests for result-only rendering, durability, and happy-path tool cards

**Out of scope:**
- **CLI-only fix** (adding `name`/`input` to `tool-call-result` wire events without Hub projection) — does not survive Hub restart or cross-window gaps alone; may be noted in discuss-phase but is not the deliverable
- **Rewriting ToolCard UI** or adding new tool-specific views — display components already exist; this phase fixes data plumbing
- **Changing Cursor Agent CLI event semantics** — HAPI adapts to what Cursor emits
- **Tool permission / approval UX redesign** — permission metadata may attach to projection when already available, but MCP approval cards are Phase 3
- **Backfill from external/non-stored sources** — projection is built from messages Hub has ingested; no migration from third-party logs
- **Multi-user or backward-compat shims** — repo policy: single user, delete over deprecate

## Constraints

- Projection merge must be idempotent: duplicate or out-of-order delivery of start/result must converge to the same final state.
- Must not increase generic `'Tool'` / `'unknown'` placeholder usage in production paths covered by acceptance tests.
- Follow package test runners: Vitest for `web/` and `cli/`, `bun:test` for `hub/` and `shared/`.
- TypeScript strict; 4-space indentation; co-located `*.test.ts` next to changed modules.
- Run GitNexus `impact` before editing shared symbols; run `detect_changes` before commit.

## Acceptance Criteria

- [ ] Hub stores a merged tool call projection per `(sessionId, callId)` that survives session refetch and Hub process restart
- [ ] Completed projection includes `name`, `input`, `status`, result/error, and start/completion timestamps sufficient for existing `ToolCard` rendering
- [ ] Raw/standardized agent messages continue to persist and replay as they do today (no removal of message history)
- [ ] Web test: result-only message window + projection → tool card shows real tool name and input (not `'Tool'`)
- [ ] Web test or integration scenario: after refresh/reconnect/pagination simulation, tool cards for known `callId`s still show real tool name and input
- [ ] `bun run typecheck` passes
- [ ] `bun run test` passes (including updated/added projection and reducer tests)
- [ ] Existing start+result timeline tests and tool grouping tests pass without behavioral regression
