## Summary

Phase 01.2 adds end-to-end durable tool call projection: Zod schema + SQLite persistence + Hub ingest wiring + Web store + SSE/pagination integration. The architecture is sound and the TDD discipline is apparent throughout. No critical bugs or security vulnerabilities found.

Three warnings surface in the Hub persistence layer: a byte-count mismatch in the projection size cap, an unguarded SQLite variable-count limit in the bulk read query, and an incomplete reconcile scan for long sessions after restart. Four info items are minor design gaps, all low-risk.

## Warnings

### WR-01: `MAX_PROJECTION_BYTES` checks character count, not byte count

**File:** `hub/src/store/toolCalls.ts:17`
**Issue:** The 64 KB cap is enforced using `json.length`, which counts JavaScript UTF-16 code units — not UTF-8 bytes. For tool inputs or outputs that contain non-ASCII characters (e.g. file content with emoji, CJK text), `json.length` underestimates the actual stored byte count. A single BMP character outside ASCII has `.length = 1` but consumes up to 3 UTF-8 bytes; a supplementary-plane character has `.length = 2` and consumes 4 UTF-8 bytes. A projection right at the 65 536-character limit could occupy up to ~196 KB on disk. The variable name `MAX_PROJECTION_BYTES` documents intent as bytes, making the discrepancy misleading.

**Fix:**
```typescript
const MAX_PROJECTION_BYTES = 65536

export function upsertToolCall(...): void {
    const json = JSON.stringify(projection)
    const byteLength = Buffer.byteLength(json, 'utf8')   // bun:sqlite is Node-compatible
    if (byteLength > MAX_PROJECTION_BYTES) {
        throw new Error(
            `Tool call projection for callId ${callId} exceeds ${MAX_PROJECTION_BYTES} bytes (got ${byteLength})`
        )
    }
    // ...
}
```

### WR-02: Unguarded SQLite variable limit in `getBySessionAndCallIds`

**File:** `hub/src/store/toolCalls.ts:43-46`
**Issue:** The function generates one `?` placeholder per element of `callIds` and spreads the entire array into `.all()`. SQLite's default `SQLITE_MAX_VARIABLE_NUMBER` is 999. If a page of messages references ≥ 1 000 distinct tool call IDs, `db.prepare(...)` throws `"too many SQL variables"`. The error propagates uncaught through `store.toolCalls.getBySessionAndCallIds` → IIFE in `MessageService.getMessagesPage` → HTTP handler, turning a full page request into a 500 for any session with a dense tool-call history.

The call site in `messageService.ts` (lines 141–145) does not have a try-catch around this path.

**Fix:** Chunk `callIds` into batches of ≤ 999 and merge the results:
```typescript
const SQLITE_MAX_VARS = 999

export function getBySessionAndCallIds(
    db: Database,
    sessionId: string,
    callIds: string[]
): Record<string, ToolCallProjection> {
    if (callIds.length === 0) return {}
    const result: Record<string, ToolCallProjection> = {}
    for (let i = 0; i < callIds.length; i += SQLITE_MAX_VARS) {
        const chunk = callIds.slice(i, i + SQLITE_MAX_VARS)
        const placeholders = chunk.map(() => '?').join(', ')
        const rows = db.prepare(
            `SELECT call_id, projection FROM tool_calls WHERE session_id = ? AND call_id IN (${placeholders})`
        ).all(sessionId, ...chunk) as Array<{ call_id: string; projection: string }>
        for (const row of rows) {
            const parsed = safeJsonParse(row.projection)
            const validated = ToolCallProjectionSchema.safeParse(parsed)
            if (validated.success) {
                result[row.call_id] = validated.data
            }
        }
    }
    return result
}
```

### WR-03: `reconcileSessionToolCalls` silently truncates to 200 messages

**File:** `hub/src/sync/toolCallProjection.ts:147`
**Issue:** `reconcileSessionToolCalls` calls `store.messages.getMessages(sessionId, 200)` with a hardcoded limit. Sessions with more than 200 messages have their older messages excluded from reconciliation. After a hub restart (before the ingest hook has processed any new messages), tool call projections from messages beyond the 200-message window are never recovered. For any result-only page load that references a historical tool call (e.g. a user scrolls far back), the projection — and thus the real tool name/input — would remain missing.

This is distinct from the ingest-path persistence (which always fires at message-arrival time and survives restart). It matters for sessions that pre-date deployment of this phase, or any session whose rows were inserted before the ingest hook was wired.

**Fix:** Use the store's full scan rather than the limited `getMessages` helper, or paginate until exhausted:
```typescript
export function reconcileSessionToolCalls(sessionId: string, store: Store): void {
    // Paginate through all messages instead of capping at 200.
    let allMessages = store.messages.getMessages(sessionId, 10_000)
    // getMessages returns at most 10k rows; for extremely long sessions
    // a cursor-based scan would be needed — acceptable starting point.
    const projectionMap = new Map<string, ToolCallProjection>()
    for (const msg of allMessages) {
        const events = extractToolCallEventsFromMessageContent(msg.content, msg.createdAt)
        for (const event of events) {
            projectionMap.set(event.callId, mergeToolCallProjection(projectionMap.get(event.callId) ?? null, event))
        }
    }
    for (const [callId, projection] of projectionMap) {
        try {
            store.toolCalls.upsert(sessionId, callId, projection)
        } catch { /* oversize / schema violation — non-critical sidecar */ }
    }
}
```

At minimum, the comment `// Load all messages for the session (use max limit)` at line 147 should be corrected to document the actual limit.

### IN-01: Empty `callId` from wire not filtered before event creation

**File:** `hub/src/sync/toolCallProjection.ts:33` and `:67`
**Issue:** `extractToolCallEventsFromMessageContent` accepts `data.callId === ''` for the Cursor wire format and `block.id === ''` for legacy blocks (both gated on `typeof ... === 'string'` without a length check). An empty-string callId passes through `mergeToolCallProjection` and `store.toolCalls.upsert` without error, then fails `ToolCallProjectionSchema.safeParse` on read-back (because `callId: z.string().min(1)`) and is silently discarded. The SSE event emitted with `callId: ''` is dropped by `SyncEventSchema.safeParse` in the web client. Net effect is a no-op, but the empty callId travels further than necessary.

**Fix:** Add a length guard at extraction:
```typescript
if (data.type === 'tool-call' && typeof data.callId === 'string' && data.callId.length > 0) { ... }
if (data.type === 'tool-call-result' && typeof data.callId === 'string' && data.callId.length > 0) { ... }
// same for legacy block.id and block.tool_use_id
```

### IN-02: SSE emitted outside try-catch — projection SSE fires even on failed upsert

**File:** `hub/src/socket/handlers/cli/sessionHandlers.ts:142-148`
**Issue:** `onWebappEvent({ type: 'tool-call-projection-updated', ... })` is called after the try-catch block, so it fires unconditionally even when `store.toolCalls.upsert` threw (e.g. oversize projection). The web client patches its in-memory store, so the current session renders correctly. But the projection was never written to SQLite. On the next cold load or reconnect, `fetchLatestMessages` returns `toolCalls: {}` for this callId, and `mergePageToolCalls` won't delete the existing in-memory entry (spread preserves unknown keys). Net result: stale in-memory projection from the SSE persists in `toolProjectionStore` until `clearProjectionsForSession` fires (session-removed event). The trade-off is documented in SUMMARY as intentional. The code comment at line 144–146 does not mention the SSE implication.

**Fix:** Either move `onWebappEvent` inside the try block (only emit when upsert succeeds), or add a comment explaining why emission is unconditional:
```typescript
// Emit regardless of upsert outcome so the live client has the projection in memory.
// If upsert failed (oversize), the projection is in-memory only until page reload.
onWebappEvent?.({ type: 'tool-call-projection-updated', sessionId: sid, callId: event.callId, projection })
```

### IN-03: `reconciledSessions` Set grows unbounded over instance lifetime

**File:** `hub/src/sync/messageService.ts:40`
**Issue:** The `private readonly reconciledSessions = new Set<string>()` is never pruned. Each unique `sessionId` ever passed to `getMessagesPage` is added and never removed (there is no session-delete hook in `MessageService`). For a long-running hub serving many short-lived sessions, the Set accumulates one string entry per session indefinitely. Memory impact is low (strings only) but the leak is structural: a 36-char UUID string × 10 000 sessions ≈ 360 KB, negligible in practice, but the pattern differs from the session-cleanup discipline elsewhere in the codebase.

**Fix:** Wire a `clearSession(sessionId: string)` method that removes from `reconciledSessions` and call it from the `session-removed` handling path in `SyncEngine`:
```typescript
clearSession(sessionId: string): void {
    this.reconciledSessions.delete(sessionId)
}
```

### IN-04: `error` field in `ToolCallProjectionSchema` is never populated by merge logic

**File:** `hub/src/sync/toolCallProjection.ts:117-138`
**Issue:** `ToolCallProjectionSchema` declares `error: z.unknown().optional()`, suggesting it should carry error metadata for failed tool calls. In practice, `mergeToolCallProjection` sets `status: 'failed'` and stores the error output in `result` — but never touches the `error` field. Any consumer checking `projection.error` always gets `undefined`. This is confusing for future contributors who may expect the `error` field to be populated for `status === 'failed'` projections.

Similarly, `status: 'pending'` is defined in the schema enum but unreachable via the merge function (new projections start at `'in_progress'`). If `'pending'` is reserved for a future use case, a comment should document that intent; otherwise the enum value is dead code.

**Fix:** Either populate `error` in the failed-result path:
```typescript
return {
    ...prev,
    status,
    result: !incoming.isError ? incoming.output : undefined,
    error: incoming.isError ? incoming.output : undefined,
    completedAt: incoming.at
}
```
Or remove the `error` field from the schema if `result` is the canonical error carrier. Document the `'pending'` status intent or remove it if unused.
