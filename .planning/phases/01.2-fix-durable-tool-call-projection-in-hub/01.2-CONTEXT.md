# Phase 01.2: Fix durable tool call projection in Hub — Context

**Gathered:** 2026-05-25
**Status:** Ready for planning

<domain>
## Phase Boundary

Hub maintains a canonical, session-scoped tool call projection keyed by `callId`, merged from ingested `tool-call` / `tool-call-result` agent payloads without altering raw message history. Web consumes that projection so tool activity cards show real tool name, input, status, and result after pagination, refresh, SSE reconnect, or result-only message windows — fixing generic `Tool` placeholders from window-local pairing alone.

</domain>

<decisions>
## Implementation Decisions

### Hub persistence & ingest (single source of truth)
- **D-01:** **Dedicated SQLite table** `tool_calls` with composite key `(session_id, call_id)` and a JSON (or normalized columns) payload for merged projection fields. Do **not** store projection inside `sessions.agent_state` — that blob is permission/request oriented and would mix concerns.
- **D-02:** **Upsert on every message ingest** in Hub (`MessageService` / store path): when standardized content includes `tool-call` or `tool-call-result`, merge into `tool_calls` idempotently (out-of-order and duplicate-safe per SPEC).
- **D-03:** **Raw messages unchanged** — projection is additive; `messages` table and existing emit paths stay as today.

### Backfill & durability
- **D-04:** **Eager, idempotent session reconcile** — Hub exposes `reconcileSessionToolCalls(sessionId)` that scans all stored messages for a session and converges `tool_calls` (same merge rules as ingest). Run on first `getMessagesPage` for that session per Hub process lifetime (or when store schema version bumps), so pre-existing sessions fix without waiting for new CLI events.
- **D-05:** Projection rows **survive Hub restart** via SQLite only (no in-memory-only cache as source of truth).

### Web delivery contract
- **D-06:** **Enrich paginated message responses** — extend `getMessagesPage` return shape with `toolCalls: Record<callId, ToolCallProjection>` containing entries for every `callId` referenced in the returned `messages` array (not the full session map on every page — keeps pages coherent with the visible window).
- **D-07:** **Do not add a separate parallel REST resource** for tool calls in v1 — one request per page preserves pagination atomicity and avoids client merge races.

### Live sync (SSE)
- **D-08:** **New `SyncEvent` discriminant** (e.g. `tool-call-projection-updated`) in `@hapi/protocol`, emitted after each Hub upsert/reconcile merge, carrying `{ sessionId, callId, projection }` (or a small batch map if planner prefers one event per ingest).
- **D-09:** Web maintains a **session-scoped projection map** in `web/src/lib/toolProjectionStore.ts` — module-private `Map<sessionId, Record<callId, ToolCallProjection>>` (same layering pattern as `messageWindowState.ts`). Merge from paginated `getMessages` `toolCalls` via `mergePageToolCalls` on each page fetch and from SSE `tool-call-projection-updated` via `patchProjection`. **Not** TanStack `queryKeys.messages` (unused; messages load through message-window store per RESEARCH A1). `message-received` alone is insufficient because result-only cards need identity before/without pairing in the window.

### Web reducer integration
- **D-10:** Extend **`reduceChatBlocks` / `reduceTimeline` context** with `projectionsByCallId: Map<string, ToolCallProjection>` — same pattern as `permissionsById`, not a pre-pass that mutates messages.
- **D-11:** On `tool-result` / result-only path, seed `ensureToolBlock` from projection `name` / `input` / `status` when present; keep `permissionsById` as **supplement** for approval metadata only, not primary tool identity.
- **D-12:** Reuse existing **`ensureToolBlock` placeholder upgrade** rules (`'Tool'` / `'unknown'`) — projection supplies non-placeholder seeds; in-window start+result happy path unchanged.

### CLI wire
- **D-13:** **Hub-only deliverable** — do not rely on adding `name`/`input` to CLI `tool-call-result` wire events for this phase. Optional future CLI enrichment is out of scope (SPEC); Hub projection remains authoritative for durability and cross-window gaps.

### Protocol & types
- **D-14:** Define **`ToolCallProjection` Zod schema** in `@hapi/protocol` (fields per SPEC req 2: `callId`, `name`, `input`, `status`, result/error payloads, `startedAt`/`createdAt`, `completedAt`, optional permission metadata when Hub has it). Hub store and Web reducer import from protocol only.

### Claude's Discretion
- Exact `tool_calls` column layout vs JSON blob; index choices.
- Whether SSE sends one projection per event or batches per ingest tick.
- `reconcileSessionToolCalls` trigger (process-lifetime set vs schema migration flag).
- Web query-key placement and cache merge helper naming.
- Hub unit vs integration test split for upsert vs reconcile.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `ensureToolBlock` — Already merges stronger `name`/`input` over placeholders; projection feeds the result-only seed path.
- `getPermissions(agentState)` — Keep for approval UX metadata; do not overload as canonical tool identity store.
- `getMessagesPage` — Natural enrichment point for per-page `toolCalls` map aligned with pagination cursors.
- `SyncEventSchema` discriminated union — Add projection-updated event alongside `message-received` / `messages-invalidated`.

### Established Patterns
- **Derived state in Hub store, not Web inference** — Matches session/message durability elsewhere (`hub/src/store/`).
- **Strict Zod at `@hapi/protocol` boundary** — New projection + SSE event types follow `SessionPatchSchema` / `SyncEventSchema` patterns.
- **Reducer context maps** — `permissionsById` precedent for `projectionsByCallId`.

### Integration Points
- Hub: message ingest → upsert `tool_calls` → optional reconcile on read → publish SSE.
- Hub: `getMessagesPage` JSON → Web pagination service → `reduceChatBlocks`.
- Web: `useSSE` → merge projection map → invalidate or patch messages query as needed.

</code_context>

<specifics>
## Specific Ideas

- User directive: optimize for **framework maintainability and logical separation**, not implementation cost.
- Architectural stance: **Hub owns canonical tool identity**; Web is a consumer; CLI wire remains thin.
- Pagination rule: ship **per-page `toolCalls`** keyed to callIds in that page’s messages, not whole-session dumps on every fetch.

</specifics>

<deferred>
## Deferred Ideas

- CLI `tool-call-result` wire enrichment (defense-in-depth only; not durable alone).
- Dedicated `GET /sessions/:id/tool-calls` REST resource (unnecessary if page enrichment + SSE suffice).
- Storing projection inside `agentState` JSON (rejected — wrong aggregate).

</deferred>
