## Key Design Decision: block-already-exists guard

When both tool-call and tool-result messages are in the window (happy path), `ensureToolBlock` is called twice for the same callId. The second call (from tool-result) would normally overwrite the first (from tool-call) with the projection name — even if the projection name differs. The fix: check `toolBlocksById.has(c.tool_use_id)` before the tool-result seed. If the block already exists (start+result both in window), pass `'Tool'` as seed name so `ensureToolBlock`'s non-placeholder upgrade logic leaves the real name intact. If the block does not exist (result-only window), use projection name.
