# Phase 02.1: Code Review Report

**Reviewed:** 2026-05-26T12:00:00Z  
**Depth:** standard  
**Files Reviewed:** 28  
**Status:** issues_found

## Summary

Reviewed phase 02.1 implementation across shared protocol, hub routes/store, CLI per-message `listSkills`, and web policy removal plus header sheet and `/` autocomplete merge. Policy removal is largely complete: no remaining `skillPolicy` writes, POST route deleted, preamble removed, and web mutations stripped.

The main functional gap is **stale skill discovery on the web composer**: `useSkills` caches forever and autocomplete never refetches after the initial load, so new skills added mid-session do not appear in `/` suggestions until the user opens the skills sheet or manually invalidates the query. That conflicts with D-14 (“latest discovery”) and the mobile workflow this phase targets.

Secondary concerns: wire coercion that defaults missing `valid` to `true`, CLI `cachedSkills` with no consumer after preamble removal, and message-queue latency when `listSkills` is slow.

## Critical Issues

### CR-01: Composer `/` autocomplete uses stale skill discovery

**File:** `web/src/hooks/queries/useSkills.ts:84-98`, `web/src/router.tsx:357-366`  
**Issue:** `useSkills` sets `staleTime: Infinity` and `getSuggestions` / `getEffectiveSkillSuggestions` only filter the in-memory query result—they never refetch. `getAutocompleteSuggestions` in the router calls `getSkillSuggestions(query)` on each `/` keystroke but still reads the same cached array. After the first fetch (session mount), newly discovered skills are invisible in autocomplete until `SkillsPolicySheet` opens (which calls `refetch`) or the user hits retry/invalidate elsewhere. CLI refreshes discovery on each sent message (`runCursor.ts:192-193`), but the web cache is not invalidated on send, so mobile composer and CLI discovery diverge.

**Fix:** Refetch (or invalidate) skills when autocomplete runs or when a message is sent—for example:

```typescript
// useSkills.ts — refetch before building suggestions when data may be stale
const getSuggestions = useCallback(async (queryText: string): Promise<Suggestion[]> => {
    if (api && sessionId) {
        await query.refetch()
    }
    const latest = query.state.data?.success && query.state.data.skills
        ? query.state.data.skills.map(normalizeSkillSummaryForWire)
        : skills
    return getEffectiveSkillSuggestions(latest, queryText)
}, [api, sessionId, query, skills])
```

Alternatively, lower `staleTime`, call `queryClient.invalidateQueries({ queryKey: queryKeys.skills(sessionId) })` from the send path in `router.tsx` / `useSendMessage`, or have `getAutocompleteSuggestions` await `refetch()` when `query.startsWith('/')`.

## Warnings

### WR-01: `normalizeSkillSummaryForWire` defaults missing `valid` to true

**File:** `shared/src/skillPolicy.ts:12`  
**Issue:** `valid: skill.valid ?? true` treats omitted `valid` as allowed. Autocomplete uses `isValidSkillForAutocomplete` (`valid === true` only). A malformed or legacy RPC payload without `valid` would be suggested even when the underlying skill is invalid. CLI `listSkills` always sets `valid` explicitly today; risk is at the hub wire boundary if a future caller omits the field.

**Fix:** Default to `false` when `valid` is missing, or require `ListSkillsResponse` parsing with `SkillSummarySchema` on the hub before normalization:

```typescript
valid: skill.valid === true,
```

### WR-02: CLI `cachedSkills` is updated but never consumed

**File:** `cli/src/cursor/runCursor.ts:149-168,192-193`  
**Issue:** Per-message `refreshSkills()` updates `cachedSkills`, but nothing reads it after preamble removal. The only use is returning the last cache on `listSkills` failure inside the same helper. Phase D-01/D-02 describe discovery refresh so skills join the “allowed set” for the turn; with no preamble and no handoff to `loop()`, it is unclear this refresh affects agent behavior (Cursor may rely on native filesystem discovery only). The call still adds latency on every user message if filesystem scans are slow.

**Fix:** Either wire refreshed skills into the agent path if still required, or drop `cachedSkills` and call `listSkills` only where results are needed (e.g. document that refresh is for RPC/cache warming only). If keeping refresh, avoid blocking the message queue on failure—fire-and-forget with logging, or timeout.

### WR-03: User messages block on `listSkills` completion

**File:** `cli/src/cursor/runCursor.ts:192-199`  
**Issue:** `onUserMessage` awaits `refreshSkills()` before `messageQueue.push`. A slow or hung filesystem walk delays every queued turn. Coalescing helps concurrent messages but not a single slow scan.

**Fix:** Start `refreshSkills()` without awaiting before push, or race with a short timeout and proceed with the last cache; ensure agent turn still gets fresh discovery via Cursor’s own mechanism if that is the actual source of truth.
