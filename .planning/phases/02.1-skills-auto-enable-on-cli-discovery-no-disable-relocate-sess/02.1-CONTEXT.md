# Phase 02.1: Skills auto-enable and header skills UI - Context

**Gathered:** 2026-05-26
**Status:** Ready for planning

<domain>
## Phase Boundary

Close the Phase 2 gap between “session skill policy” UX and the desired mobile behavior:

1. **Discovery = availability** — Each time the CLI prepares a user turn, refresh skill discovery; every **valid** skill returned is available for the agent and composer (no user disable, no persisted policy map).
2. **Header skills panel** — Move session skills UI from the chat composer to `SessionHeader` (beside Files), read-only name list with Global/Local scope tags; invalid skills show a red **Invalid** i18n tag.
3. **Composer autocomplete** — Skill suggestions use **`/`** prefix (not `$`); remove composer skills-policy affordance.
4. **Remove policy machinery** — Delete `metadata.skillPolicy`, Hub `POST /sessions/:id/skill-policy`, CLI skill-policy preamble, and all tri-state / enforcement UI tied to session policy.

Out of scope: editing skill files, global Cursor config, MCP policy (Phase 3), changing `/settings/skills` catalog into a policy editor (remains read-only inventory with richer metadata).

</domain>

<decisions>
## Implementation Decisions

### Discovery refresh & allowed set (D-01–D-03)
- **D-01:** **Refresh trigger (user choice B):** Before each user message is queued in `runCursor`, call `listSkills(workingDirectory)` (or await a single in-flight refresh) so newly discovered skills join the allowed set on the next turn — not only at runner startup.
- **D-02:** **Allowed set definition:** All skills with `valid === true` from the latest discovery result are allowed for that turn. No `skillPolicy` map, no `enabled` / `disabled` / `inherited` rows.
- **D-03:** **Invalid skills:** Excluded from prompt filtering and `/` autocomplete; may still appear in the session header list with a red **Invalid** badge (see UI).

### Remove skillPolicy & Hub API (D-04–D-07) — user choice A
- **D-04:** **Remove `skillPolicy` from `@hapi/protocol`** (`MetadataSchema`, `SessionPatchSchema`, types) and from Hub session persistence / merge (`sessionMergeService`, resume/handoff). Bump SQLite schema version with fail-fast migration policy per repo rules (no backward compat — drop column/field).
- **D-05:** **Delete Hub route** `POST /sessions/:id/skill-policy` and `createSkillPolicyRoutes`; remove Web `ApiClient` skill-policy mutation methods and TanStack optimistic policy updates.
- **D-06:** **Delete CLI policy preamble path:** Remove `skillPolicyPreamble.ts`, `applySkillPolicyToFormattedMessage` usage in `runCursor`, and related tests. **No replacement preamble** (user choice 6B).
- **D-07:** **Delete shared policy helpers** where only used for policy (`isEffectivelyAllowed`, policy branches in `isSkillSuggestible`) — replace with **valid-only** discovery checks for autocomplete; keep `normalizeSkillSummaryForWire`.

### Header skills UI (D-08–D-12) — user choice 3A + 4
- **D-08:** **Entry:** New icon button in `SessionHeader`, placed **next to Files** (before Outline when present). Tooltip and `aria-label`: **Skills** (i18n key e.g. `session.skills.title`).
- **D-09:** **Surface:** Reuse bottom **Dialog sheet** pattern from current `SkillsPolicySheet` (mobile-safe), opened from header — **not** inside Files route, **not** in composer.
- **D-10:** **List layout:** **Flat** rows — skill **name** only as primary text; one small scope pill: **Global** (`source: user`) or **Local** (`source: project`). No description, invocationMode, pathHint, tri-state, reset, browse-from-sheet footer policy actions, or enforcement badges.
- **D-11:** **Invalid row:** Still listed in session panel; show red **Invalid** pill (i18n e.g. `session.skills.invalidTag`) in addition to Global/Local when `valid === false`. Do not show `invalidReason` body text in this panel (keeps “names + tags only”).
- **D-12:** **Remove from composer:** Delete `showSkillsPolicyButton`, `SkillsPolicySheet` mount under `HappyComposer`, `SkillPolicyRow`, `SkillTriStateControl`, `EnforcementBadge` (if only used here), and policy mutation props on `SessionChat` / `useHappyComposerState`.

### Composer `/` skill autocomplete (D-13–D-14) — user choice 5A + `/`
- **D-13:** **Trigger prefix:** Skill autocomplete activates when composer query starts with **`/`** (replace all `$` skill paths in `useSkills`, `router.tsx` autocomplete gate, `useHappyComposerHandlers` suggestion handling).
- **D-14:** **Suggestions:** All **valid** skills from latest `useSkills` discovery query for the session; suggestion insert text uses `/skill-name` form (parallel to old `$name` keys). No policy filter.

### Settings catalog (D-15)
- **D-15:** **`/settings/skills`** unchanged in role: read-only catalog with descriptions/validity for browse. Update copy that referenced “session policy” toggles. Scope labels in settings may adopt Global/Local for consistency (planner discretion).

### i18n (D-16)
- **D-16:** Replace User/Project session labels with **Global** / **Local** under `session.skills.*`. Add **`session.skills.invalidTag`** (and sheet title `session.skills.title` = “Skills”). Remove or repurpose keys: `composer.skillsPolicy`, `session.skills.enforcement.hapi`, `session.skills.inheritedHelper`, reset/tri-state strings.

### Claude's Discretion
- Exact Skills header icon SVG (sparkles vs wand) matching 18×18 header icons.
- Whether to keep search in sheet when skill count > 8 (likely yes; search by name only).
- Settings page Global/Local rename scope (session panel is mandatory).
- Schema version number and migration one-liner in hub store.
- Component rename: `SkillsPolicySheet` → `SessionSkillsSheet` (recommended, not required).

</decisions>
