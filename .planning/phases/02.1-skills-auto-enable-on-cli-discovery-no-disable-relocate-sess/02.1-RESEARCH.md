# Phase 02.1: Skills auto-enable and header skills UI - Research

**Researched:** 2026-05-26
**Domain:** Remove session skill policy; CLI per-turn discovery refresh; header read-only skills sheet; `/` composer autocomplete
**Confidence:** HIGH for deletion scope and file inventory; HIGH for `/` vs slash-command collision; MEDIUM for merged-autocomplete UX details

## Summary

Phase 02.1 is a **gap-closure refactor** of Phase 2, not a greenfield feature. The codebase already has full skill discovery (`listSkills` вҶ’ `GET /sessions/:id/skills` вҶ’ `useSkills`), a policy stack (`metadata.skillPolicy`, Hub `POST /skill-policy`, CLI preamble, composer `SkillsPolicySheet`), and `$`-prefixed skill autocomplete. Locked CONTEXT decisions **delete** the policy stack and **relocate** the session skills UI to `SessionHeader`, while making **every valid discovered skill** implicitly allowed with **per-user-message CLI refresh** and **no preamble**.

The highest-risk planning item is **D-13 (`/` skill prefix)**: slash commands already use `/` via `useSlashCommands` and `router.tsx` routes non-`$` queries to slash suggestions. Replacing `$` with `/` for skills **without** merging lists will break either skill or slash autocomplete. The planner must specify a **merged `/` autocomplete** (skills + slash commands, deduped by key, labeled by kind) or an explicit disambiguation rule вҖ” the user chose `/` for skills, not removal of slash commands.

SQLite `skillPolicy` lives inside the **`metadata` JSON column**, not a dedicated column. Per repo fail-fast policy (`hub/src/store/index.ts` `SCHEMA_VERSION = 12`), bumping version forces DB recreation; stripping orphaned `skillPolicy` keys from old rows is optional hygiene, not required for correctness once code ignores the field.

**Primary recommendation:** One vertical slice per tier вҖ” (1) shared schema + delete policy helpers/routes, (2) CLI refresh + delete preamble, (3) Web header sheet + merged `/` autocomplete + composer cleanup вҖ” with tests updated in the same commit wave as each deletion.

### Pattern 1: Single-flight skill refresh before queue (CLI)

**What:** Coalesce concurrent `listSkills` calls when the user sends messages quickly.  
**When:** Every `onUserMessage` in `runCursor` (D-01).  
**Example:**

```typescript
// Pattern: single-flight refresh (planner implements in runCursor.ts)
let cachedSkills: SkillSummary[] = [];
let refreshInFlight: Promise<SkillSummary[]> | null = null;

function refreshSkills(): Promise<SkillSummary[]> {
    if (!refreshInFlight) {
        refreshInFlight = listSkills(workingDirectory)
            .then((skills) => {
                cachedSkills = skills;
                return skills;
            })
            .catch((err) => {
                logger.debug('[cursor] listSkills failed; using last cache', err);
                return cachedSkills;
            })
            .finally(() => {
                refreshInFlight = null;
            });
    }
    return refreshInFlight;
}

session.onUserMessage(async (message, localId) => {
    await refreshSkills();
    const formattedText = formatMessageWithAttachments(/* ... */);
    // D-06: no preamble вҖ” push formattedText directly
    messageQueue.push(formattedText, enhancedMode, localId);
});
```

### Pattern 2: Merged `/` autocomplete (Web)

**What:** When query starts with `/`, return **both** skill and slash-command suggestions.  
**Why:** `useSlashCommands` already treats `/` as its prefix [VERIFIED: `web/src/hooks/queries/useSlashCommands.ts` lines 51вҖ“54]. `router.tsx` currently sends `$` вҶ’ skills, else вҶ’ slash [VERIFIED: `router.tsx` lines 356вҖ“360].  
**Example:**

```typescript
// router.tsx вҖ” illustrative merge (dedupe by suggestion.key)
const getAutocompleteSuggestions = useCallback(async (query: string) => {
    if (!query.startsWith('/')) {
        return await getSlashSuggestions(query); // non-/ legacy paths if any
    }
    const [skills, commands] = await Promise.all([
        getSkillSuggestions(query),
        getSlashSuggestions(query),
    ]);
    const seen = new Set<string>();
    return [...skills, ...commands].filter((s) => {
        if (seen.has(s.key)) return false;
        seen.add(s.key);
        return true;
    });
}, [getSkillSuggestions, getSlashSuggestions]);
```

Update `toSuggestion` in `useSkills.ts` to emit `key`/`text`/`label` as `` `/${skill.name}` `` (not `$`). Update `useHappyComposerHandlers` `markSkillUsed` to detect `/` prefix [VERIFIED: currently checks `$` line 64].

## Common Pitfalls

### Pitfall 1: `/` prefix collision with slash commands

**What goes wrong:** Skill autocomplete steals `/` or slash commands stop appearing.  
**Why it happens:** Phase 2 used `$` for skills specifically to avoid this collision [VERIFIED: `router.tsx`]. User chose `/` for skills in 02.1 without removing slash commands.  
**How to avoid:** Merged suggestion list with distinct `description` or `source` field; document ordering (e.g. skills before commands). Add tests in `useSkills.test.ts` and router-level autocomplete test.  
**Warning signs:** `getAutocompleteSuggestions` branches `if (query.startsWith('/')) return skillsOnly`.

### Pitfall 2: Stale Web discovery vs fresh CLI allow set

**What goes wrong:** Header/autocomplete show old skill list after user adds `SKILL.md` on disk.  
**Why:** `useSkills` uses `staleTime: Infinity` [VERIFIED: `useSkills.ts` line 89].  
**How to avoid:** `refetch()` when opening header sheet; optional `queryClient.invalidateQueries(queryKeys.skills(sessionId))` after focus; CLI refresh (D-01) covers agent turns, not Web cache.  
**Warning signs:** User adds skill, sees it in CLI turn but not in sheet until reload.

### Pitfall 3: Incomplete policy deletion (strict schema / SSE)

**What goes wrong:** Typecheck passes but runtime still accepts `skillPolicy` or UI shows tri-state.  
**How to avoid:** Delete in one wave: shared schema вҶ’ hub engine вҶ’ web API вҶ’ UI; grep `skillPolicy|SkillPolicy|skill-policy` until clean. Update `useSessionActions.test.ts`, `SkillsPolicySheet.test.tsx` (delete or rewrite).  
**Warning signs:** `MetadataSchema` still has optional `skillPolicy`; fixtures still POST `/skill-policy`.

### Pitfall 4: SCHEMA_VERSION bump without dev communication

**What goes wrong:** Hub fails to start against existing DB.  
**How to avoid:** Bump version per D-04; plan step: remove `~/.hapi` DB or documented path; in-memory tests unaffected.  
**Warning signs:** `Expected 13, found 12` on hub boot.

### Pitfall 5: `getSkillSourceLabelKey` still maps userвҶ’"User"

**What goes wrong:** UI shows User/Project instead of Global/Local (D-16).  
**How to avoid:** New keys `session.skills.scope.global` / `.local` (or rename existing `source.user`/`source.project` strings in en + zh-CN).
