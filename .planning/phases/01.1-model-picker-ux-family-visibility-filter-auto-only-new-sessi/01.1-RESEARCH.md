## User Constraints (from CONTEXT.md)

### Locked Decisions

#### Visibility filter
- **D-01:** Visibility granularity is **model family** (e.g. Opus 4.7), not each raw variant id.
- **D-02:** Storage: `localStorage`, key `hapi-visible-model-families` — string array of stable family keys.
- **D-03:** Unconfigured default: empty/missing key = **all families visible** until user saves a filter in Settings.
- **D-04:** Settings entry: one row on Settings index → dedicated **`/settings/models`** sub-route (Variant A drill-down; no inline mega-section on index).
- **D-05:** Optional `hapi-visible-model-families-configured: true` when user has ever saved; distinguish "all visible" default from explicit empty selection if needed.

#### New session
- **D-06:** Remove `ModelSelector` from new session; `spawnSession` omits `model` (Auto).
- **D-07:** Discovery still powers Settings and composer; discovery failure UX matches Phase 1 patterns.

#### Composer picker
- **D-08:** Replace flat `HappyComposerOverlays` model list with nested picker: primary menu (Auto + enabled families) + Edit → Options (Thinking, Fast, Context, Effort).
- **D-09:** Auto always available in composer; not part of family filter.
- **D-10:** Runtime submit: compose valid raw id from family + options; send only `model` string (no separate effort API fields).
- **D-11:** Footer link: "Manage visible models…" → `/settings/models`.
- **D-12:** Trigger shows family + option summary (e.g. "Opus 4.7 · Medium · Thinking").

#### Model families
- **D-13:** Input: `CursorModelSummary[]` from `getCursorModels(machineId)`; parse label/id into family key, display name, variants.
- **D-14:** Options UI enables only combinations that exist in discovered variants for that family.
- **D-15:** Machine-scoped discovery × global visibility: effective list = `families(discovered) ∩ visibleFamilies` (or all if filter unset).

#### Non-goals
- **D-16:** No separate Hub/CLI fields for effort / `modelReasoningEffort` beyond display metadata.
- **D-17:** No static model catalog in protocol — discovery remains runtime-owned via `agent models`.
- **D-18:** No per-machine visibility profiles in v1.

#### UAT / Phase 1 polish
- **D-19:** Update Phase 1 UAT items 2–3 when implementing (new session Auto; explicit model via in-session only).
- **D-20:** CURS-02 launch wording: model selection moves to in-session only.

### Claude's Discretion
- Exact family parser heuristics and slug algorithm.
- Component split (`ModelPickerOverlay` vs refactor `HappyComposerOverlays`).
- i18n key naming under `settings.models.*` and `composer.modelPicker.*`.

### Deferred Ideas (OUT OF SCOPE)
- Server-side family metadata in protocol (v2 if needed).
- Per-machine visibility profiles.
- Variant B/C sketch patterns (bottom sheet, inline expand on settings index).

## Summary

Phase 01.1 is a **Web-only UX slice** on top of the Phase 1 runtime config contract. Discovery, spawn, and in-session model switch plumbing already exist: `agent models` → CLI `parseCursorModelList` → hub route → `api.getCursorModels(machineId)` → `useCursorModels` → flat options in `router.tsx` / `HappyComposerOverlays`. [VERIFIED: codebase] The phase replaces flat raw-id lists with **family grouping**, adds a **global localStorage visibility filter**, removes launch-time model UI, and implements a **two-panel composer picker** that still calls `onModelChange(rawId)` with a single string.

Live CLI output on the research machine confirms the `id - label` shape and the naming patterns the parser must handle (Codex effort/fast suffixes, Opus/Sonnet versioned families, Composer 2, Auto). [VERIFIED: local CLI] Family key derivation and option composition are the main implementation risk; everything else follows existing hooks, routes, and Phase 1 discovery error UX.

**Primary recommendation:** Add `cursorModelFamilies.ts` + `useVisibleModelFamilies.ts` first, then `/settings/models` and composer `ModelPickerOverlay`, then delete new-session model UI and update tests/UAT. Keep hub/protocol unchanged; intersect `groupModelsIntoFamilies(discovery)` with visible keys at render time in Web only.

### Pattern 1: Family grouping module (`cursorModelFamilies.ts`)

**What:** Pure functions over `CursorModelSummary[]`.

**Types (suggested):**

```typescript
export type ModelFamily = {
    key: string           // stable slug, e.g. "claude-opus-4-7"
    displayName: string // e.g. "Opus 4.7"
    variants: CursorModelSummary[]
}

export type ModelOptionSelection = {
    effort?: 'low' | 'medium' | 'high' | 'xhigh' | 'max'
    thinking?: boolean
    fast?: boolean
    context1m?: boolean
}
```

**When to use:** Settings checklist, composer picker, StatusBar label.

### Pattern 2: Family parsing approach (verified samples)

**Input shape:** `{ id: string, label?: string }` from discovery. [VERIFIED: `shared/src/schemas.ts`]

**Live CLI samples (2026-05-25):** [VERIFIED: local CLI]

| id | label | Inferred family display | Option signals |
|----|-------|-------------------------|----------------|
| `auto` | Auto | (excluded from families) | — |
| `composer-2` | Composer 2 | Composer 2 | — |
| `composer-2-fast` | Composer 2 Fast | Composer 2 | fast |
| `gpt-5.3-codex-high-fast` | Codex 5.3 High Fast | Codex 5.3 | effort=high, fast |
| `claude-opus-4-7-thinking-medium-fast` | Opus 4.7 1M Medium Thinking Fast | Opus 4.7 | 1M, medium, thinking, fast |
| `claude-4.6-opus-high-thinking` | Opus 4.6 1M Thinking | Opus 4.6 | 1M, high, thinking |
| `claude-4.6-sonnet-medium` | Sonnet 4.6 1M | Sonnet 4.6 | 1M, medium |

**Recommended algorithm (planner discretion within D-13):**

1. **Drop** `id === 'auto'` from family grouping (D-09).
2. **Family key:** Prefer id-prefix heuristic, normalized:
   - `claude-opus-4-7-*` → key `claude-opus-4-7`
   - `claude-4.6-opus-*` → `claude-4.6-opus`
   - `claude-4.6-sonnet-*` → `claude-4.6-sonnet`
   - `gpt-{version}-codex-*` → `gpt-5.3-codex` (strip effort/thinking/fast suffixes from key)
   - `composer-2*` → `composer-2`
   - Fallback: slugify first two label tokens (`displayName` → key).
3. **Display name:** Prefer label tokens before `1M` / effort words: e.g. `Opus 4.7`, `Codex 5.3`, `Composer 2`.
4. **Per-variant flags** (for Options panel — enable only if ≥1 variant matches):
   - `thinking`: id includes `-thinking` or label contains `Thinking`
   - `fast`: id ends with `-fast` or label contains `Fast`
   - `context1m`: label contains `1M` or id contains `[1m]` (legacy)
   - `effort`: extract from id (`-low-`, `-medium-`, `-high-`, `-xhigh-`, `-max-`) or label (`Low`, `Medium`, `High`, `Extra High`, `Max`)
5. **composeVariantId(family, selection):** Filter `family.variants` by flags; tie-break deterministically (e.g. prefer exact effort match, then fewest extra suffixes). Return `null` if no match (disable Apply / show invalid).
6. **decomposeModelId(id, families):** Reverse map for StatusBar + picker initial state when session already has raw id.

**Confidence:** HIGH for Opus/Codex/Composer patterns on research machine; MEDIUM for future vendor ids — cover with fixture tests from `agent models` captures, not static catalog.

## Common Pitfalls

### Pitfall 1: Confusing “unset filter” vs “hide all families”

**What goes wrong:** Empty `localStorage` array treated as “show nothing” on first visit.

**How to avoid:** D-03 + D-05 — only treat empty array as “hide all” when `hapi-visible-model-families-configured === 'true'`.

### Pitfall 2: Composer shows raw ids after grouping

**What goes wrong:** StatusBar/picker still display `claude-opus-4-7-thinking-medium-fast`.

**How to avoid:** Always render via `decomposeModelId` + i18n option labels (D-12).

### Pitfall 3: Options enable impossible combinations

**What goes wrong:** User selects Thinking + Fast + Max when no variant exists.

**How to avoid:** D-14 — disable toggles with zero matching variants; disable confirm until `composeVariantId` non-null.

### Pitfall 4: New session still fetches models

**What goes wrong:** Wasted RPC and flaky tests after removing selector.

**How to avoid:** Remove `useCursorModels` from `NewSession`; keep discovery on settings + session routes only (D-06, D-07).

### Pitfall 5: Breaking Phase 1 model switch semantics

**What goes wrong:** Closing overlay before `setModel` completes confuses switch state.

**How to avoid:** Keep `handleModelChange` in `useHappyComposerHandlers` (closes overlay, haptic); don’t bypass `SessionChat` switch state machine.

### Pitfall 6: Stale tests / UAT

**What goes wrong:** CI expects model `<select>` on new session.

**How to avoid:** Update `NewSession.test.tsx`, `01-UAT.md` tests 2–3 (D-19).
