<objective>
Close UAT Test 3 (CURS-03 / D-06): make the in-session Cursor model switch truthful end-to-end. The next `agent` spawn after a switch MUST run with the newly selected `--model` while reusing the same `--resume <session_id>`, the Web composer must stop optimistically flipping the big model label, and the Web copy must say "Applies next message" instead of "Applies next run" since remote mode re-spawns `agent` per message.

Purpose: 01-09 wired the UI selector and Hub metadata persistence for in-session model switch and 01-06 shipped the status-bearing apply contract, but the CLI runner side never adopts the new model — `currentModel` is captured as a `const`, `applyCursorSessionConfig` returns `applies-next-run` for the model branch without mutating anything, `CursorSession.model` is `readonly`, and `cursorRemoteLauncher.runMainLoop` reads `session.model` instead of `batch.mode.model` from the queue. The Phase 1 spike at `agent-transcripts/f150696b` already proved `agent -p ... --resume <same_session_id> --model <new>` switches the live model on the next turn; this plan exercises that path and aligns the Web composer/locale copy with remote-mode "per-message" reality.

Output: A mutable CLI runtime model carrier (`currentModel` + `syncModel` callback + `CursorSession.setModel`), a remote launcher that prefers `batch.mode.model` over `session.model` (with `session.model` retained as fallback only when `mode.model` is undefined), a composer status surface that keeps the BIG label tied to the model that answered the most recent turn (the captured `previousModel`) while the pending target is shown as a secondary "Applies next message" label, updated en + zh-CN locale strings, and regression tests on both runners (Vitest in CLI/Web) that lock the behavior.
</objective>
