<objective>
Close the active and inactive set-session-config success-like acknowledgement gap (CURS-02 / WR-01) so unsupported effort-only and modelReasoningEffort-only payloads no longer return applied/applies-next-run on either runtime branch.

Purpose: 01-11 + 01-14 + 01-15 removed the spawn boundary, persistence write surface, and CLI launch contract for unsupported effort, but the public runtime config response shape still claims success on the runner RPC handler and on the Hub inactive branch. Verifier truth #2 fails because callers cannot distinguish "applied" from "ignored". This plan flips both responses to a typed failed CursorRuntimeConfigApplyResult and proves no persistence occurs.

Output: `applyCursorSessionConfig` returns `status: 'failed'` (echoing current model/effort) for effort-only / modelReasoningEffort-only payloads with no permission-mode change; `SyncEngineSession.applySessionConfig` returns the same failed shape on the inactive branch when `model` and `permissionMode` are absent; `hasRuntimeConfigRequest` no longer references `effort` or `modelReasoningEffort`; flipped + new tests prove the contract.
</objective>

<interfaces>
Existing runtime facts:
- `applyCursorSessionConfig(payload, state)` accepts `state.currentModel`, `state.currentModelReasoningEffort`, `state.currentEffort` and currently returns `{ applied: { permissionMode } }` when `hasModelConfigRequest === false`. The `currentModel`/`currentModelReasoningEffort`/`currentEffort` slots already exist on the state object so a failed result can echo them without new plumbing.
- `CursorRuntimeConfigApplyResultSchema` (in `shared/src/schemas.ts`) accepts `status: 'applied' | 'applies-next-run' | 'failed'` with `model`, `modelReasoningEffort`, `effort`, and `reason` fields.
- `SyncEngineSession.applySessionConfig` builds `hasRuntimeConfigRequest` as `config.model !== undefined || config.modelReasoningEffort !== undefined || config.effort !== undefined`. After this plan it must be `config.model !== undefined` only.
- `SyncEngineSession.applySessionConfig` inactive branch currently returns `applies-next-run` whenever `hasRuntimeConfigRequest` is true, even if no supported field is present. After this plan, an effort-only inactive payload (no `model`, no `permissionMode`) must return `status: 'failed'` and skip `sessionCache.applySessionConfig`.
- Active branch behavior for `model` / `permissionMode` is owned by 01-14 and must remain unchanged by this plan.
</interfaces>

<preconditions>
01-14 and 01-15 are complete (persistence + CLI launch contract closed). This plan only adjusts the response shape and the inactive request-classification predicate; it does not introduce new persistence surface or new effort schema fields.
</preconditions>
