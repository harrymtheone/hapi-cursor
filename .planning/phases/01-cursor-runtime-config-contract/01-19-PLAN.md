<objective>
Close UAT Test 5 part 1 (CURS-04 / D-13): the mobile session list must keep converging on idle/thinking/running/waiting/error/completed status for ALL sessions even while the user is inside one chat session. Drop the `{ sessionId: selectedSessionId }` narrowing branch in `web/src/App.tsx`'s `eventSubscription` so the global session-list status feed stays alive at all times.

Purpose: `web/src/App.tsx:274-279` currently narrows the SSE subscription to `{ sessionId: selectedSessionId }` when a session page is open. The Hub then filters `session-updated` events to that one session id, so the rest of the session list stops receiving status/completion patches and the spinners freeze at whichever `statusKind` was last cached. Per `.planning/debug/session-list-spinner-stuck.md`, this is the root cause of "切换到其他 chat session，spinner 不会自动切换".

Output: A single-file production change replacing the narrowed-subscription branch with an unconditional `{ all: true }` subscription, plus two regression tests — one in `web/src/App.test.tsx` (created if absent) locking the new memoization shape, and one in `web/src/hooks/useSSE.test.tsx` proving that a cross-session `session-updated` patch updates the correct SessionList cache entry regardless of the selected session.
</objective>

- The Hub side (per `.planning/debug/session-list-spinner-stuck.md`) filters `session-updated` events by the subscription scope: a `{ all: true }` subscription receives every session's patches; a `{ sessionId: X }` subscription receives only session X's patches.

- `useSSE.patchSessionDetail(sessionId, patch)` mutates `queryClient.queryData<SessionResponse>(queryKeys.session(sessionId))`. The open-session detail view continues to be patched correctly even when the subscription is `{ all: true }` (the detail cache is keyed by session id, not by subscription scope).
